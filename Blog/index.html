Test-Driven Development

5-5-15

I decided to do a little research on test-driven development because I've realized I like the DBC challenges that require us to pass each test before moving on to the next section of our code. In test-driven development the first thing that's done is writing a test that fails but sets a goal of what you want the code to accomplish. The code should ideally be written in as few lines and as refactored as possible.

The key to test-driven development is knowing what you want your program to do. You need to be able to understand what your end result should look like before starting. Once you realize what you want you can write a test that will pass once that specific code is implemented. Once you write the code that passes the test, you move on to another test to accomplish a new programming goal. And so on.

Test-driven development is great for working out a problem step by step. Moving through the code one test at a time can help pin point what code is working perfectly, and what code is breaking the program (b/c of course you shouldn't move on with your programming until the test you're working on passes). Ideally your end code will be cleaner and drier using this technique because you are only writing the minimum code to pass each of your tests.

OOP Concepts

Classes v. Modules

4-22-15

I wasn't (and still am not) very familiar with modules, which is the reason I wanted to Blog about them today. I've been pretty curious about classes v. modules myself, so I when I saw that on the challenge I knew they must be pretty similar. And it seems they are. Classes and modules are both used to group a bunch of code together, code that you will often want to type when building a giant program. They are tools to help you group specific lines of code together and easily use them over and over again.

My understanding is modules can be used as a separate storage compartment when you need it. Classes are used to create instances but modules are not. Modules can collect methods and constants (and even classes!) but modules aren't used for instances.

Modules can be used to store pieces of code that can then be used across multiple classes, let's say. If you look up modules (like I had to), you will often read that this helps prevent name clashes. If two methods or even two classes got named the same thing, modules can store them differently so your program knows which one to call on at the right time. Just think of modules as another layer of which to group and store your code.

Taking A Look At Map

An Enumerable Object In Ruby

4-9-15

Map is an interesting enumerable method and one that seems like it could/should be used quite often. The simple basics of map it to take an array and run every item in that array through a block of code, similar to what .each does. Map is different however because map keeps the items in an array, and map! replaces the original array with the newly mapped array.

I used map in an earlier challenge this week during the insert commas into numbers challenge. I had different groups of individual numbers,four different numbers 1 2 3 4, and I needed it read as 1234. In order to push the numbers together I used .map{|y| y.join("")}. This took every number as y and ran the method join on them, and since I joined at ("") I took out the spaces between the numbers. It was the first time I had ever used map, and I think that's about the simplest way to use map.

Obviously map can be used for a lot of other things, but it's important to keep in mind that map will keep your array items in array form. I say this because a lot of times map will just modifiy the items in the array that you are planning on calling upon later, chances are if you modify those items you'll want to use the modified versions instead of their original items. This is where map! comes in b/c you actually overwrite the original array with the mapped array. An example I found online while researching map! used an array full of names, but all the names were in lower case. To capitalize all the names in the array you could use names.map! {|x| x.capitalize}. Now the items in the array names have been changed from lower case to upper case. The array with lower case names is now gone forever and replaced with an array with upper case names (NOTE: inside the {} you can use any variable you want. I've used y and now x in my examples in this blog, but it doesn't matter what you call them). I have a feeling map! is likely to be more common than map as we continue down our programming path.

Arrays and Hashes

Understanding the Difference in Ruby

April Fools Day, 2015

I made a small tweak to the blog template so hopefully this works. More tweaks to come tomorrow. Arrays and hashes can both be useful tools in Ruby because instead of storing and assigning one variable at a time, A+H's let you store multiple variables at once. The difference is in how these values are stored.

An array stores variables in an ordered fashion and saves each variable as a number (starting with 0) based on the order you store them in the array. Because of this an array is great for storing an ordered list of items.

A hash on the other hand pairs the variables with a key and a value. Basically, each variable (value) can be named (key) so you can easily call upon that variable without having to know which place in the order it lands. I'm still in the very early stages of using both arrays and hashes, but they seem to be so similar that once you get the hang of using one, the other should be extremely easy to pickup.

CSS Positioning

The Difference Between Absolute, Relative, and Fixed Positioning

3-27-15

Positioning elements is one of the most important features to CSS, but can also be extremely confusing. In fact, I wrote this blog post b/c I was confused about positioning myself. Here I will go through the differences between Absolute, Relative, and Fixed positioning. There's a fourth type called Static, but that is the default position for any element, and is rarely ever used in CSS.

The first type of positioning is Absolute. Absolute positioning allows you to place an object wherever you want on the screen. The coordinates you type in are based on the screen. So if you type in

Selector {

position: absolute;

top: 0px;

left: 0px;

}

the element will be place in the top left corner of your screen. Absolute positioning can be great and easy, but it also takes elements out from the flow of the page, meaning it's difficult to use with other objects. This is where Relative Positioning comes in

Relative Positioning places an object relative to itself and where it fits normally on the page. If you type in the same code as above but use relative instead of absolute, the object won't move at all because you told it to move from its top 0px and from its left 0px.

Fixed Positioning essentially glues an object to the page so when you scroll the object an still be seen. It is great if you want a navagational bar to always be seen, but it can be dangerous because you don't want a fixed object to cover up other objects as you scroll down the page. Each type of position has its own pros and cons and figuring out which position is the best all depends on what type of design you want for your site.

Week 1 DBC

Learning Git

3-18-15

Git is a type of version control that helps its users keep track of work flow. Git helps you save your changes as you go, so you will always have safe spots to fall back on should you make a mistake with your code in the future. Git also works well when multiple people are working on the same project. Git helps those people work on different tasks at the same time without worry of working on an outdated file. Git allows branches of the tree to be modified, without any worry of the trunk being destroyed.

Anytime a change is made to a file, it will show up in your git status. Typing git status into your terminal will bring up a few sections. If your file is new, you will see it in the Untracked Files section. If you made a change to an existing file, you will see it in the Changes Not Staged For Commit section. Git provides instructions for each section on how to go about sending these files to the Changes To Be Committed section. Think of committing as a final confirmation before pushing these files to the web. You can only push files after they have been committed. Once pushed, they will appear remotely on the web, in this case GitHub.

GitHub allows users to send their work to the web easily by linking up with the Git on their computer. With GitHub, Bob in TN can work on a project with Lynn in VA and they both can get updates by pushing and pulling their work to/from GitHub. Git helps you keep track of your changes by easily accessing a log by typing git log in your terminal. Your git log shows every change that has been pushed to GitHub.
